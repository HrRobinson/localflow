# saiifeguard G1 — standalone crate — implementation plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the standalone `saiifeguard` Rust crate — the command-guard engine,
the pack loader + TOML schema, the four v1 packs, and the `test` / `explain` /
`check` CLI — with zero coupling to the TypeScript app. Fail-open, hard latency
cap, allow-before-deny. G2 (saiife integration) is a separate plan.

**Architecture:** A Cargo workspace at `guard/` producing one binary,
`saiifeguard`. Engine pipeline: parse → normalize → pre-filter → match
(allow-before-deny) → decide. Packs are TOML (metadata + allow/deny regex
lists, each deny carrying a reason), loaded from built-in defaults and, at
runtime, from `guard/packs/*.toml`. The crate never touches the network and has
no dependency on the app. See `docs/superpowers/specs/2026-07-14-saiifeguard-design.md`.

**Tech Stack:** Rust (edition 2021, pinned via `rust-toolchain.toml`),
`serde` + `toml` (pack parsing), `regex` (RE2, no backtracking), `serde_json`
(hook payload), `clap` (CLI). `cargo test` for unit + golden-corpus tests.

## Global Constraints

- **Public repo:** no personal notes / names / maintainer references in any
  committed file. MIT-compatible dependencies only.
- Conventional Commits, subject ≤ 50 chars; PR title likewise (hard commitlint).
- `guard/target/` is gitignored; never commit build artifacts.
- **Fail-open is inviolable:** every error path (bad JSON, oversize input,
  regex timeout, pack-load failure, panic) resolves to ALLOW + a stderr warning.
  No code path may block on the guard's own malfunction.
- **Allow-before-deny:** allow-patterns always evaluated first and short-circuit.
- **Standalone:** the crate must build and test with `cargo test` alone, no
  Node, no app. Ship the four packs as built-in defaults so the CLI works with
  no external files.
- Idiomatic Rust: `Result` + `thiserror`, no `unwrap` in library paths (tests
  may `unwrap`), `#![forbid(unsafe_code)]`.
- TDD: write the failing test first, watch it fail, then implement.

### Task 5: Engine — match with allow-before-deny + decision type

**Files:**
- Create: `guard/crates/saiifeguard/src/engine.rs`
- Test: inline.

**Interfaces:**
- Produces:
  - `enum Decision { Allow, Deny { pack: String, rule: String, reason: String } }`
    plus a trace variant/field for `explain` (which allow short-circuited, or
    that the pre-filter skipped a pack).
  - `pub struct Engine { packs: Vec<Pack>, prefilter: Prefilter }` with
    `pub fn evaluate(&self, raw_cmd: &str) -> Decision`.
  - Evaluate order: normalize → per pack, pre-filter gate → allow rules (match ⇒
    ALLOW, stop) → deny rules (first match ⇒ DENY). No match anywhere ⇒ ALLOW.

- [ ] **Step 1: Failing tests** using a small hand-built pack:
  a deny pattern matches ⇒ `Deny` with pack+rule+reason; an allow pattern that
  also matches a deny ⇒ `Allow` (allow wins); a safe command ⇒ `Allow`;
  first-of-two denies is the one reported.

- [ ] **Step 2: Implement.** Build the combined pre-filter from all packs'
  triggers. Per pack, skip if its triggers exclude the command.

- [ ] **Step 3: Verify** `cargo test engine`. Commit.

```bash
git commit -am "feat: engine allow-before-deny matcher"
```

### Task 6: Hook payload parsing + inline `-c` fallback + fail-open + latency cap

**Files:**
- Create: `guard/crates/saiifeguard/src/payload.rs`
- Modify: `guard/crates/saiifeguard/src/engine.rs` (inline-payload re-entry, cap)
- Test: inline.

**Interfaces:**
- Produces:
  - `pub fn command_from_hook_json(input: &str, max_bytes: usize) -> Result<String, PayloadError>`
    — extract the command from a PreToolUse-shaped JSON payload; oversize or
    malformed ⇒ `Err` (caller fails open).
  - `pub fn inline_payloads(normalized: &str) -> Vec<String>` — trigger-regex
    extraction of `bash -c '…'` / `sh -c "…"` / `python -c '…'` inner strings.
  - Engine change: after judging the outer command, also judge each extracted
    inline payload; a deny in any inner payload denies.
  - A latency cap around evaluation: exceeding the budget ⇒ ALLOW + warn.

- [ ] **Step 1: Failing tests.** `command_from_hook_json` extracts the command
  from a representative Claude Code PreToolUse Bash payload; an oversize input
  (> cap) ⇒ `Err`; malformed JSON ⇒ `Err`. `inline_payloads("bash -c 'rm -rf /'")`
  yields `["rm -rf /"]`. Engine: `bash -c 'git reset --hard'` ⇒ `Deny`.

- [ ] **Step 2: Implement.** Keep the JSON shape tolerant (extract the Bash
  command field; unknown shapes ⇒ `Err` ⇒ fail open). Cap the input size before
  parsing. Guard the evaluate call with the latency budget.

- [ ] **Step 3: Verify** `cargo test payload`. Commit.

