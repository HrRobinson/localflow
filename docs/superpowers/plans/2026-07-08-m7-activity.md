# M7 — Activity view + Overview stats Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A plain-language **Activity feed page per session** — the vibe-coder
lens over the same hook events that drive the status colors — plus the
**Overview stats strip** (counts by status + oldest unattended wait), folded
into this milestone because it reads the same data with the same glanceability
spirit. No sparklines, no charts, no persistence.

**Architecture:** Main gains one small piece of plumbing: each `SessionManager`
record keeps an **in-memory ring of its last 200 activity entries** — every
hook event it applies plus the lifecycle moments it already knows (created,
reopened, closed, exited, environment moved) — each entry `{ timestamp, kind,
status }`. It also stamps `needsYouSince` on `SessionInfo` when a session enters
`needs-you` and clears it on leaving, both inside `setStatus` using the
manager's overridable clock. Two IPC surfaces expose the ring: `activity:get`
(pull the whole ring) and a pushed `activity:event` channel (mirrors the
existing `onStatus` push). The ring is **not persisted** — the feed is honest
about starting fresh each launch. The renderer keeps all copy in **pure,
unit-testable modules** (`activity-format.ts` for event→language + relative
time, `overview-stats.ts` for the counts derivation); a new `Activity` view
(App view union + sidebar nav + session switcher) and a stats strip on the
Overview render them. The "waiting Nm" fragment and the Activity view's "open
terminal" button reuse the existing `nextNeedsYou` + `openSession` machinery so
jumping to attention behaves exactly like `cmd+u`.

**Tech Stack:** Electron main/preload/renderer, React 19, vitest
(`tests/unit/**`, node env), Playwright `_electron` e2e driving hook events via
the existing `endpoint.json` POST fixture.

Spec: `docs/superpowers/specs/2026-07-08-m7-activity-design.md` (approved 2026-07-08).

## Global Constraints

- Conventional Commits, subject ≤ 50 chars, body lines ≤ 100 chars (husky + CI
  commitlint; PR titles ≤ 50 chars). `npm run check` (eslint + prettier +
  typecheck + vitest) green before every commit.
- This milestone runs in a **separate git worktree**: Task 1 Step 1 begins with
  `npm ci` (a fresh worktree has no `node_modules`). Every later task assumes it
  ran.
- The activity ring is **per-session, in-memory, capped at the last 200
  entries**, and is **never persisted** — a restored session starts with an
  empty ring, and the UI says "since saiife started". The `saveSessions`
  pick list in `index.ts` is NOT extended (no `activity`, no `needsYouSince`).
- Each ring entry is exactly `{ timestamp: number, kind: ActivityEventKind,
  status: SessionStatus }`. `kind` is one of the three applied hook events
  (`UserPromptSubmit` / `Notification` / `Stop`) or a lifecycle moment
  (`created` / `reopened` / `closed` / `exited` / `moved`). `timestamp` and all
  clocks use `SessionManager`'s `now()` override so tests are deterministic.
- `needsYouSince` is set on entering `needs-you` and cleared on leaving it
  (including on `exited`), **inside `setStatus`**, using the manager's `now()`.
  It lives on `SessionInfo` (in-memory only), so it flows to the renderer via
  the existing `list()` poll — a restored session (always `exited`) has none.
- Restored sessions (`restore` / `restoreBrowser`) record **no** activity entry:
  "created" would be a lie for something loaded from disk. Reopening a restored
  session records `reopened` as its first entry.
- Browser panes (`kind: 'browser'`) get a ring of lifecycle events only (they
  never receive hook events — `applyHookEvent` already early-returns for them);
  the Activity page states this.
- All human-facing activity copy and time formatting live in **one renderer
  module** (`activity-format.ts`) so wording is reviewable/translatable in one
  place. The relative-time and duration helpers are **pure with an injected
  `now`** for tests.
- Overview stats are **numbers only** — counts by status in the compact form
  `N working · N needs you · N done · N off`, plus the oldest unattended
  `waiting Nm`. No charts, no sparklines, no per-turn analytics.
- Out of scope (spec §7): sparklines, flow visualization, per-turn analytics,
  event persistence across restarts, exporting. Nothing in this plan builds any
  of it.
- **Parallel-branch note (M6):** `App.tsx`'s view union and `Sidebar.tsx`'s nav
  are ALSO edited by a concurrent M6 (Changes view) branch. Keep those two
  diffs minimal and purely additive (add the `'activity'` union member, one nav
  button, one render branch — touch nothing else there). The Overview stats
  strip lives in `Landing.tsx`, which M6 does not touch — no conflict there.
  See Self-Review Notes for the merge expectation.
- DOM/e2e contract (new stable hooks): `.overview-stats`, `[data-stat="<status>"]`,
  `.stats-waiting` (Overview); `.activity-switcher`, `.activity-current`,
  `.activity-line`, `.activity-open`, `.activity-note` (Activity view). Existing
  contracts (`.pane[data-pane-id][data-status]`, `.session-row[data-session-id]`,
  `.session-status`, `[data-nav-session]`, `[data-nav-environment]`) are unchanged.

  /** Append one entry to a session's ring (capped), notifying listeners. */
  private recordActivity(id: string, kind: ActivityEventKind): void {
    const rec = this.sessions.get(id)
    if (!rec) return
    const entry: ActivityEntry = { timestamp: this.now(), kind, status: rec.info.status }
    rec.activity.push(entry)
    if (rec.activity.length > ACTIVITY_LIMIT) rec.activity.shift()
    this.activityCbs.forEach((cb) => cb(id, entry))
  }

  /** The session's activity ring, oldest first. Empty for unknown ids. */
  getActivity(id: string): ActivityEntry[] {
    return this.sessions.get(id)?.activity.map((e) => ({ ...e })) ?? []
  }
```

import type { ActivityEntry, SessionInfo } from '../../../shared/types'

/**
 * All human-facing activity copy lives here (one place to review/translate,
 * per the spec). Each function is pure; the two time helpers take an injected
 * `now` so they are trivially unit-testable and tick off the renderer's normal
 * re-render (the 1s session poll) without their own timers.
 */

/** Plain-language copy for one entry — the view appends the relative time. */
export function activityLine(kind: ActivityEntry['kind']): string {
  switch (kind) {
    case 'created':
      return 'session created'
    case 'reopened':
      return 'session reopened'
    case 'closed':
      return 'you closed the terminal'
    case 'exited':
      return 'process exited'
    case 'moved':
      return 'moved to another environment'
    case 'UserPromptSubmit':
      return 'you sent a prompt'
    case 'Notification':
      return 'waiting for your approval'
    case 'Stop':
      return 'turn finished'
  }
}

/** "just now" / "Ns ago" / "Nm ago" / "Nh ago" / "Nd ago". */
export function relativeTime(then: number, now: number): string {
  const s = Math.floor(Math.max(0, now - then) / 1000)
  if (s < 5) return 'just now'
  if (s < 60) return `${s}s ago`
  const m = Math.floor(s / 60)
  if (m < 60) return `${m}m ago`
  const h = Math.floor(m / 60)
  if (h < 24) return `${h}h ago`
  return `${Math.floor(h / 24)}d ago`
}

/** Compact elapsed span "45s" / "12m" / "3h" / "2d" for the "waiting for N" copy. */
export function humanDuration(ms: number): string {
  const s = Math.floor(Math.max(0, ms) / 1000)
  if (s < 60) return `${s}s`
  const m = Math.floor(s / 60)
  if (m < 60) return `${m}m`
  const h = Math.floor(m / 60)
  if (h < 24) return `${h}h`
  return `${Math.floor(h / 24)}d`
}

/**
 * The persistent header line so the page is glanceable without reading history.
 * needs-you shows how long the user has kept the agent waiting; the other
 * states get an honest one-liner.
 */
export function currentStateLine(session: SessionInfo, now: number): string {
  switch (session.status) {
    case 'needs-you':
      return typeof session.needsYouSince === 'number'
        ? `⏳ waiting for your approval for ${humanDuration(now - session.needsYouSince)}`
        : '⏳ waiting for your approval'
    case 'working':
      return '● working'
    case 'idle':
      return '✓ idle — last turn finished'
    case 'running':
      return '● running'
    case 'exited':
      return '○ exited'
  }
}

import type { SessionInfo, SessionStatus } from '../../../shared/types'

export interface StatSegment {
  status: SessionStatus
  label: string
  count: number
}

export interface OverviewStats {
  /** Non-zero counts in a fixed, glanceable order. */
  segments: StatSegment[]
  /** Longest current wait (ms) from needsYouSince, or null when nobody waits. */
  oldestWaitMs: number | null
}

// Fixed display order + the compact label per status. `running` sits between
// needs-you and done; when there are none it simply drops out, reproducing the
// canonical "N working · N needs you · N done · N off" form from the spec.
const ORDER: { status: SessionStatus; label: string }[] = [
  { status: 'working', label: 'working' },
  { status: 'needs-you', label: 'needs you' },
  { status: 'running', label: 'running' },
  { status: 'idle', label: 'done' },
  { status: 'exited', label: 'off' }
]

/** Pure derivation for the Overview stats strip. Numbers only — no charts. */
export function deriveOverviewStats(sessions: SessionInfo[], now: number): OverviewStats {
  const counts = new Map<SessionStatus, number>()
  let oldest: number | null = null
  for (const session of sessions) {
    counts.set(session.status, (counts.get(session.status) ?? 0) + 1)
    if (session.status === 'needs-you' && typeof session.needsYouSince === 'number') {
      oldest = oldest === null ? session.needsYouSince : Math.min(oldest, session.needsYouSince)
    }
  }
  const segments = ORDER.filter(({ status }) => (counts.get(status) ?? 0) > 0).map(
    ({ status, label }) => ({ status, label, count: counts.get(status) ?? 0 })
  )
  return { segments, oldestWaitMs: oldest === null ? null : Math.max(0, now - oldest) }
}
```

import { useEffect, useState } from 'react'
import type { ActivityEntry, SessionInfo } from '../../../shared/types'
import { activityLine, relativeTime, currentStateLine } from '../lib/activity-format'

interface Props {
  sessions: SessionInfo[]
  activeId: string | null
  onOpenSession: (id: string) => void
}

/**
 * The activity feed: a plain-language, glanceable lens over the same hook
 * events that drive the status colors — not a replacement for the terminal
 * (one click away via "open terminal"). The feed itself is main's in-memory
 * ring (activity:get) plus live pushes (activity:event); the persistent header
 * reads the session's live status/needsYouSince from the polled `sessions`
 * prop, so "waiting for N" ticks on the 1s poll.
 */
export default function Activity({ sessions, activeId, onOpenSession }: Props): React.JSX.Element {
  const [selectedId, setSelectedId] = useState<string | null>(activeId ?? sessions[0]?.id ?? null)
  const [entries, setEntries] = useState<ActivityEntry[]>([])

  // Keep the selection valid as sessions come and go (render-time adjustment,
  // the same pattern Landing/Sidebar use for stale ids).
  const valid = selectedId !== null && sessions.some((s) => s.id === selectedId)
  const currentId = valid ? selectedId : (sessions[0]?.id ?? null)
  if (currentId !== selectedId) setSelectedId(currentId)

  // Load the ring for the selected session and stream new entries for it.
  useEffect(() => {
    if (currentId === null) {
      setEntries([])
      return
    }
    let cancelled = false
    void window.saiife.getActivity(currentId).then((list) => {
      if (!cancelled) setEntries(list)
    })
    const off = window.saiife.onActivity((id, entry) => {
      if (id === currentId) setEntries((prev) => [...prev, entry])
    })
    return () => {
      cancelled = true
      off()
    }
  }, [currentId])

  const session = sessions.find((s) => s.id === currentId) ?? null
  const now = Date.now()

  if (!session) {
    return (
      <div className="activity mx-auto flex w-full max-w-[720px] flex-1 flex-col gap-4 px-6 py-8">
        <p className="m-0 text-[13px] text-gray-500">No sessions yet.</p>
      </div>
    )
  }

  return (
    <div className="activity mx-auto flex w-full max-w-[720px] flex-1 flex-col gap-4 overflow-auto px-6 py-8 text-left">
      <div className="flex items-center gap-3">
        <select
          className="activity-switcher bg-surface-raised focus:border-working rounded-md border border-white/[0.14] px-2.5 py-2 text-[13px] text-gray-200 outline-none"
          value={session.id}
          onChange={(e) => setSelectedId(e.target.value)}
          aria-label="Session"
        >
          {sessions.map((s) => (
            <option key={s.id} value={s.id}>
              {s.name}
            </option>
          ))}
        </select>
        <button
          className="activity-open cursor-pointer rounded-md border border-white/10 bg-white/[0.07] px-3 py-2 text-[13px] text-gray-200 hover:bg-white/[0.13] hover:text-white"
          onClick={() => onOpenSession(session.id)}
          onMouseDown={(e) => e.preventDefault()}
        >
          {session.kind === 'browser' ? 'open browser' : 'open terminal'}
        </button>
      </div>

      <div
        className="activity-current bg-surface-raised rounded-lg border border-white/10 px-4 py-3 text-[15px] text-gray-100"
        data-status={session.status}
      >
        {currentStateLine(session, now)}
      </div>

      {session.kind === 'browser' && (
        <p className="activity-note m-0 text-[13px] text-gray-500">
          Browser panes have no status feed — lifecycle events only.
        </p>
      )}

      <p className="m-0 text-[11px] tracking-[0.06em] text-gray-600 uppercase">
        Activity · since saiife started
      </p>

      {entries.length === 0 ? (
        <p className="m-0 text-[13px] text-gray-500">No activity yet.</p>
      ) : (

                key={entries.length - 1 - i}

                <span>{activityLine(e.kind)}</span>
                <span className="text-gray-500">· {relativeTime(e.timestamp, now)}</span>
              </li>
            ))}
        </ul>
      )}
    </div>
  )
}

**Known risks / accepted trade-offs:**
- The `needsYouSince` fast-path lag: an `onStatus` push flips a pane to needs-you
  before the next 1s poll delivers the freshly-stamped `needsYouSince`, so for up
  to ~1s the Overview "waiting Nm" / Activity header can read the fallback
  ("waiting for your approval" with no duration). `currentStateLine` and
  `deriveOverviewStats` both degrade gracefully (fallback line / excluded from
  oldest-wait) rather than showing a wrong number.
- `running` is surfaced as its own stats bucket (not dropped, not folded into
  "working"): "counts by status" must not silently lose sessions. When there are
  zero running sessions the strip reproduces the spec's exact `working · needs
  you · done · off` form; the running bucket only appears when it's non-zero.
  **(Spec ambiguity resolved:** the example form omits `running`; I kept it as an
  explicit bucket rather than hiding a status.)
- The Activity view's session switcher is self-contained rather than mirroring
  "M6's Changes view" verbatim, because M6 is an unlanded parallel branch. It's a
  plain `<select>` over all sessions (the Activity lens is global, like the
  Overview). **(Spec ambiguity resolved.)**
- Restored sessions record no activity, so a freshly-launched app shows "No
  activity yet" for untouched restored sessions until the first real event — this
  is the honest consequence of "since saiife started" (unit-tested).
- The `setState`-during-render selection guard in `Activity.tsx` follows the
  existing Landing/Sidebar stale-id pattern; it's the codebase's accepted idiom
  for adjusting state to changed props before commit.
